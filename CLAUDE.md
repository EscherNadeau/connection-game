# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

"The Connection Game" — a movie/TV trivia game where players link two corners of cinema through shared cast credits (person ↔ movie/show they acted in; titles never link to titles, people never to people). Single-page static site, vanilla JS, no framework, no build step, no dependencies, no tests.

## Running it

Open `index.html` directly in a browser (`Start-Process index.html`) — file:// works. All data comes live from the TMDB API; the app prompts for a key on first run (v3 key or v4 bearer token, stored in localStorage as `tmdb_key`). Without a valid key nothing beyond the key screen is testable.

## Files

## Architecture

**Screens** are sibling `<section id="screen-NAME">` elements toggled by `show(name)` / `showT(name)` (the latter wraps the change in a View Transition radial reveal expanding from the last pointer-down; it returns a promise that resolves when the DOM has switched — `startGame` awaits it before measuring the viewport). New screens must be added to the `screens` array.

**Items** are `{ key, id, type: 'movie'|'tv'|'person', name, img }` where `key` is `"type-id"` (e.g. `"movie-603"`). Keys are the universal currency: graph nodes, credit sets, ban lists, and challenge links all use them. `fetchItemByKey(key)` rehydrates a full item from TMDB.

**Connection checking** (`connects(a, b)`): fetches credit sets via `getConnections` into `creditsCache` (Map of key → Set of connected keys). Checks both directions because TMDB's TV aggregate credits and person combined credits can disagree. The hint system's bridge-finding intersects these cached sets, so it gets smarter as more items are placed.

**Random endpoint pools**: titles come from `/discover` sorted by `vote_count.desc` (deliberately NOT `/popular`, which is traffic-driven and gets obscure fast), people from `/person/popular`, bucketed into obscurity bands (`OBSCURITY_BANDS` page ranges). Each band keeps its own prefetched pool; `takeRandomItem` consumes from it and refills below `POOL_LOW_WATER`. The home-screen poster-rain background snapshots these pools (no extra API calls).

**Game state** lives in the single `game` object. Three modes share it: `"classic"` (connect start to goal, win = path exists), `"knowledge"` (blitz: name direct connections of one center item before the clock; no win-by-path — `checkWin` early-returns and the round ends via `timeUp` → `showKnowledgeResults`, which reuses the win modal shell and rewrites its eyebrow/h1 — `checkWin` must restore them), and `"hybrid"` — the **Double Feature** (UI name; internal mode id stayed `"hybrid"`, as did `body.hybrid-intent` and `settings.hybridN`): one start, 1–3 goals picked VISIBLY at setup via the poster deck (`goalSlots[]` + `.df-card`s — front card selected (`dfSel`), others fanned behind; one shared reroll/search acts on the front card; the deck's start display shares `data-slot="start"` with the classic card, so slot renders go through `renderSlotDisplays`), all on the board from move 1, win = every goal connected (`hybridCheckWin` → `highlightPaths` golds all winning paths; the win stat counts nodes that served 2+ paths). `game.endKey` tracks the first unreached goal so the hint machinery works unchanged; header goal chips (thumb + ellipsized name, ✓ when reached) are tappable marquee peeks; hybrid wins hide `#btn-copy-challenge` — classic/knowledge/finale win paths must un-hide it.

**Board** = pan/zoom viewport (`view` transform) + force-directed physics (`sim` Map, `physicsTick` in a permanent rAF loop gated by `boardActive`). Physics has capped repulsion + springs + a hard positional minimum-separation pass (`MIN_SEP`) that runs after integration — overlap fixes belong there, not in the force model. Start/goal nodes are `fixed`; dragged nodes win all collisions.

**Challenge links** serialize a game into the URL hash `#c=<base64url JSON>` (unicode-safe b64 helpers). Blob v1: `{v, s, g, h, t}` (+ `m:"k"`, `n` for knowledge — there `n` = challenger's bar). Blob v2 adds builder rules: `b` (budget), `bans[]`, `wps[]` (waypoints, ≤3), `ty` (allowed types as "mtp" chars), `ti`/`by` (title/creator), `par`. Blob v3 (what the Studio emits) is a *feature*: `sc[]` of scenes (`{m, s, g?, t?, n?, b?, wps?, h?, ty?}` — in v3 knowledge scenes `n` = target count, NOT bar; `h`/`ty` are per-scene hints/types, with top-level `h`/`ty` read as fallbacks for early v3 links), global `bans`, `ti`/`tg`/`by` (title/tagline/creator), `par[]` (per-scene creator scores). Link length is a non-concern by decision — blobs move to a backend (Supabase) later. `tryLoadChallenge()` runs at boot and after key entry; it must stay backward-compatible with all blob shapes. Applied rules land in `game.rules` and are enforced strictly (banned/type-blocked placements rejected in `tryPlace`, budget exhaustion = fail, waypoints must all be on the winning path in added order, par compared on win). Challenge settings mutate `settings` in memory only — never call `saveSettings()` for them.

**Quests** (the Studio runtime): every shared link is wrapped in the `quest` object (1–5 rounds) and lands on `screen-premiere` (title card + fanned posters + Roll Film). `questMulti()` (>1 round) gates the multi-scene flow: round ends route to `endRound()` → intermission modal (`#round-modal`) → `startRound(i+1)` → `showFinale()`, which merges every scene's nodes/edges (accumulated in `quest.nodes/edges`) onto one board and reuses the win-modal shell as the credits. Single-round quests keep the classic win/lose/results modals but still premiere and route quit/play-again back to the premiere. Quick play sets `quest.active = false` and quits back to the setup screen. Knowledge rounds can carry a `target` (`game.target`) — reaching it ends the round as a success before the clock.

**The Studio** (`screen-build`) is a WYSIWYG premiere: the screen reuses the premiere-card look (marquee title/tagline/credit as borderless inputs, poster fan in `#build-posters` grown from the scenes) and the scene list renders with the same `sceneSummary()` the player's premiere uses. Scenes are always explicit (`builder.scenes[]`, 1–`MAX_SCENES`=5; no implicit single-round fallback — Test/Release stay disabled at 0 scenes). Editing happens on `screen-scene`, a full screen modeled on the quick-play setup screen: mode chips under the title, the start/goal poster cards dead center with reroll/search, and ALL rule knobs (clock/target/hints/budget/allowed/via) behind a rules pill that shows a live summary (`sceneRulesSummary`) and opens `#scene-rules-sheet` (same pattern as the bans pill). Knowledge mode hides classic-only bits via a `k-mode` class toggled on BOTH `#screen-scene` and `#scene-rules-sheet` (generic `.k-mode .only-classic` / `.only-knowledge` rules). It edits a *draft* (the loose `builder.mode/start/goal/...` fields): draft changes don't void the par; only save / cut / reorder / global-rule changes do. `builder.editing` is the scene index being edited (-1 = new; new scenes auto-roll a fresh matchup). Save Scene stays disabled until the matchup is complete; ← back compares against the snapshot taken on open (`sceneSnapshot0`) and asks before discarding a dirty draft. Reorder/cut live on the scene rows on the stage (`data-act` tool buttons), not in the editor. Hints and allowed types are per-scene draft knobs in the editor; only bans are global, living in `#rules-sheet` (applied live, voids par). A Test Screening plays the whole feature through the premiere with `builderTest = true`; finishing stamps `builder.par` (single) or `builder.parRounds[]` (multi). Any material change voids the par (`voidPar()` clears both). Quitting a test run must reset `builderTest` AND `quest.active`.

**The casting call** (`screen-cast`) is the full-page browse tier (dropdowns stay the default; in-game search never uses it). Entered via `.browse-cast` buttons on endpoint cards or the home switcher ("The Back Lot"); `cast.ctx` carries the context: `{mode:"slot", slot}` (quick play — tap a card, slot fills, return to `mode` screen), `{mode:"studio", target}` (casting session — target chips Start/Goal/Via/Ban assign-and-stay, assigned cards lit via `updateCastMarks`/`data-role`, ✓ done returns to `scene` and re-renders the editor), or `{mode:"research"}` (the Back Lot — tap opens `#detail-sheet`: credits as walkable `conn-card`s, breadcrumb `cast.trail`, "Play from here" seeds a classic game). Grid cards are built from RAW TMDB results (kept in `cast.results`) because fame badges need `vote_count`/`popularity` and people show `known_for` — `makeItem` only on assign. Empty query falls back to `/trending`; typed-endpoint results lack `media_type`, so type resolution is `raw.media_type || cast.type`. `cast.seq` guards stale responses.

**The Box Office** (`screen-office`, ghost button on home) browses features as shelves of cards: `STAFF_PICKS` (curated v3 blobs bundled in app.js), Your Filmography (localStorage `"shelf"` — auto-saved by Studio Release via `saveToShelf(buildStudioBlob())`), Ticket Stubs (localStorage `"stubs"` — written by `recordStub()` from every quest ending: classic win/lose, knowledge results, budget fail, finale; skips quick play and `builderTest`). Cards boot through `loadFeature(blob, link?)` — the extracted core of `tryLoadChallenge` that ALL feature entry paths must share. "Now Showing" is reserved naming for the future daily; "The Charts" for the community shelf. **Tickets**: `printTicket()` (Studio) draws a movie-ticket PNG on canvas and embeds the blob in a `tEXt` chunk (`pngAddText`/`pngReadText`, keyword `connection-game`); redeem via the Box Office button or dropping a PNG anywhere (`redeemTicket` → `loadFeature`). Metadata-only by decision — original file required, recompressed copies lose the data; the barcode is decorative until a v2 QR.

**Settings** (`settings` object, localStorage `"settings"`): difficulty presets bundle obscurity/hints/timer; hand-tweaking any knob flips difficulty to `"custom"`.

## UI conventions

- Layout stability is a hard requirement: panels that appear/disappear (custom knobs, builder chip tray) reserve their space or scroll internally — the page must never grow and pop a scrollbar when toggling options.
- Theming is via CSS custom properties on `:root` / `body.dark`; any new color must work in both themes.
- Escape all TMDB-sourced strings with `esc()` before injecting into HTML.
- Animations: `prefers-reduced-motion` must be respected (poster rain, view transitions already do).
- The design language is "warm paper cinema": grain texture (`--grain`), pill buttons/chips, `Instrument Serif` italics for accent words (`<em>`), gold for path highlights, green `--start` / clay `--goal` for endpoints.
