# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

"The Connection Game" — a movie/TV trivia game where players link two corners of cinema through shared cast credits (person ↔ movie/show they acted in; titles never link to titles, people never to people). Single-page static site, vanilla JS, no framework, no build step, no dependencies, no tests.

## Commands

- **Run locally**: open `index.html` in a browser (`Start-Process index.html`) — file:// works.
- **Syntax check**: `node --check app.js` (the only "lint" this project has; run it after every JS edit).
- **Deploy**: `git push` to `main` — GitHub Pages (repo EscherNadeau/connection-game) redeploys in under a minute. Live at https://eschernadeau.github.io/connection-game/.
- After structural HTML changes, verify every `#id` referenced in app.js still exists in index.html: **all listeners attach at script load, so one missing element crashes the entire app silently.**

All data comes live from TMDB. A house key ships in `DEFAULT_TMDB_KEY` (deliberately public — embed decision 2026-06-12) so the app boots straight to home; a personal key saved via "change api key" (localStorage `tmdb_key`, v3 key or v4 bearer token) overrides it.

## Files

- `app.js` — all logic (~3000 lines, sectioned by `// ===== Name =====` / `// ---- Name ----` banner comments)
- `style.css` — all styles (sectioned by `/* ===== name ===== */`)

## Architecture

**Screens** are sibling `<section id="screen-NAME">` elements toggled by `show(name)` / `showT(name)` (the latter wraps the change in a View Transition radial reveal from the last pointer-down; it returns a promise that resolves when the DOM has switched — game starters await it before measuring the viewport). New screens MUST be added to the `screens` array. Current set: key, home, how, build, scene, cast, office, mode, game, premiere.

**Items** are `{ key, id, type: 'movie'|'tv'|'person', name, img, fame }` where `key` is `"type-id"` (e.g. `"movie-603"`). Keys are the universal currency: graph nodes, credit sets, ban lists, and challenge links all use them. `fetchItemByKey(key)` rehydrates a full item from TMDB. `fame` is raw TMDB popularity (people) / vote_count (titles), `null` when the source response lacked it; `fameTier(fame, type)` maps it onto the one fame vocabulary (famous/known/deep cut/crazy — shared with the casting-page badges) and returns `null` for unknown, which never scores either way. Person thresholds are calibrated to TMDB's post-2024 reworked popularity (compressed, trending-driven — megastars baseline ~5-12; see the comment on `fameTier` before touching them).

**TMDB access**: every request funnels through `tmdb()` — a concurrency gate (max 8 in flight) with automatic 429 retry honoring Retry-After. Never `fetch` TMDB directly.

**Connection checking** (`connects(a, b)`): fetches credit sets via `getConnections` into `creditsCache` (Map of key → Set of connected keys). Checks both directions because TMDB's TV aggregate credits and person combined credits can disagree. The hint system's bridge-finding intersects these cached sets, so it gets smarter as more items are placed.

**Random endpoint pools**: titles from `/discover` sorted by `vote_count.desc` (deliberately NOT `/popular`, which is traffic-driven and gets obscure fast), people from `/person/popular`, bucketed into obscurity bands (`OBSCURITY_BANDS` page ranges). Each band keeps its own prefetched pool; `takeRandomItem(excludeKey, type, level)` consumes and refills below `POOL_LOW_WATER`, recycling `servedKeys`/`usedPages` when a band drains. The home-screen poster rain snapshots these pools (no extra API calls).

**Game state** lives in the single `game` object. Three modes share it:
- `"classic"` — connect start to goal; win = path exists (BFS over `game.edges`); score = shortest path. Wins are also **star-rated by the path's links** (deep-cut scoring, TODO #17): a link's tier is its more obscure end (`edgeTier` — famous↔famous = obvious, either end deep-cut/crazy = the pull); ⭐ connected / ⭐⭐ no famous↔famous link / ⭐⭐⭐ clean + a deep/crazy link, plus a "one take" flourish when `placed` = bridge count. Badges render on the win-path arrows, not the posters. `#win-stars` is unhidden/filled ONLY by the classic win path — knowledge results, hybrid wins, and the finale credits all hide it (same shell-restore contract as the eyebrow/h1).
- `"knowledge"` — blitz: name direct connections of one center item before the clock. No win-by-path: `checkWin` early-returns and the round ends via `timeUp` → `showKnowledgeResults`, which reuses the win-modal shell and rewrites its eyebrow/h1 — the classic win path must restore them.
- `"hybrid"` — UI name **Double Feature** (internal id, `body.hybrid-intent`, and `settings.hybridN` kept the old name): one start, 1–3 goals picked visibly at setup via the poster deck (`goalSlots[]` + `.df-card`s; `dfSel` = front card; one shared reroll/search/browse acts on the front card; the deck's start display shares `data-slot="start"` with the classic card, so slot renders must go through `renderSlotDisplays`). All goals are on the board from move 1; win = every goal connected (`hybridCheckWin` → `highlightPaths` golds all paths at once; win stat counts nodes serving 2+ paths). `game.endKey` always points at the first unreached goal so the hint machinery works unchanged. Header goal chips are tappable "marquee peeks" (top billed names). Hybrid wins hide `#btn-copy-challenge` — classic/knowledge/finale win paths must un-hide it. (The original two-phase hybrid was built and cut the same day — see TODO #3 for why.)

**Undo**: `game.lastPlaced` tracks the most recent placement; `undoLast()` unwinds it (nodes/edges both directions/sim/DOM/`placed--`, refunding budget) until the next placement lands. Classic/hybrid only — `updateUndoBtn()` (piggybacked on `updateStats()`, plus explicit calls on win/over paths) hides `#btn-undo` in knowledge mode and once `won`/`over`. A hybrid undo re-runs `hybridCheckWin()` because it can un-reach a goal chip.

**Board camera**: `fitBoard()` auto-fits after each placement/undo — a short rAF "chase" (nodes are still settling, so it eases toward a moving target, capped at scale 1) that only engages when the padded bounds exceed the viewport. Any manual pan/pinch/wheel/node-drag calls `cancelFitFollow()` — the player always wins.

**Board** = pan/zoom viewport (`view` transform) + force-directed physics (`sim` Map, `physicsTick` in a permanent rAF loop gated by `boardActive`). Capped LOCAL repulsion (fades to zero at `REPULSE_RANGE` — global push straightens chains into lines; local push lets them fold into clumps) + springs + weak centering gravity (only beyond `GRAV_FREE` of the centroid, so it never fights separation) + a hard positional minimum-separation pass (`MIN_SEP`) after integration — overlap fixes belong there, not in the force model. Start/goal nodes are `fixed`; dragged nodes win all collisions and are painted inside the pointermove handler (waiting for the next tick reads as cursor lag). Tapping a node (<8px pointer travel) opens `#peek-modal` — poster/meta/overview via `loadDetail`, deliberately no credits mid-game.

**Challenge links** serialize a game into the URL hash `#c=<base64url JSON>` (unicode-safe b64 helpers). Blob v1: `{v, s, g, h, t}` (+ `m:"k"`, `n` = challenger's bar for knowledge). v2 adds builder rules: `b` (budget), `bans[]`, `wps[]` (≤3 waypoints), `ty` (allowed types as "mtp" chars), `ti`/`by`, `par`. v3 (what the Studio emits) is a *feature*: `sc[]` of scenes (`{m, s, g?, t?, n?, b?, wps?, h?, ty?}` — in v3 knowledge scenes `n` = target count, NOT bar; per-scene `h`/`ty` with top-level fallbacks for early v3 links), global `bans`, `ti`/`tg`/`by`, `par[]`. Blob `m:"h"` is reserved for double-feature links (not yet implemented; share buttons hidden in that mode). Backward compatibility with ALL blob shapes is mandatory. Applied rules land in `game.rules` and are enforced strictly (banned/type-blocked placements rejected in `tryPlace`, budget exhaustion = fail, waypoints must all be on the winning path, par compared on win). Challenge settings mutate `settings` in memory only — never call `saveSettings()` for them. Link length is a non-concern by decision — blobs move to a backend (Supabase) later.

**Now Showing (the daily)**: date-seeded (FNV-1a → mulberry32) double bill of English-original titles from discover pages 1-8 — deterministic across players, cached per-date in localStorage `dailyPuzzle`. `#daily-strip` on home → `pendingDaily` → consumed by `startGame` into `dailyActive`; classic `checkWin` records the FIRST completion of the day to `dailyLog`, shows the 🔥 streak, and unhides `#btn-share-daily` (spoiler-free share text). `#win-daily`/`#btn-share-daily` follow the same win-modal shell-restore contract as `#win-stars`.

**`loadFeature(blob, link?)` is the single boot path for features** — URL links (`tryLoadChallenge`), Box Office cards, ticket redemption, and future community rows all go through it. It populates `quest` (incl. `quest.blob` for replays) and shows the premiere.

**Quests** (the feature runtime): every loaded feature lands on `screen-premiere` (title card + fanned posters + Roll Film). `questMulti()` (>1 round) gates the multi-scene flow: round ends → `endRound()` → intermission modal → `startRound(i+1)` → `showFinale()`, which merges every scene's nodes/edges (accumulated in `quest.nodes/edges`) onto one board and reuses the win-modal shell as the credits. Single-round quests keep the normal win/lose modals but route quit/play-again back to the premiere. Quick play sets `quest.active = false`. Knowledge rounds can carry a `target` — reaching it ends the round early as a success.

**The Studio** (`screen-build`) is a WYSIWYG premiere: marquee title/tagline/credit as borderless inputs, poster fan grown from the scenes, scene rows rendered with the same `sceneSummary()` the player's premiere uses. Scenes are always explicit (`builder.scenes[]`, 1–`MAX_SCENES`=5; Test/Release/Print stay disabled at 0 scenes). Reorder/cut live on the scene rows (`data-act` tool buttons), not in the editor. Editing happens on `screen-scene`, a full screen modeled on the quick-play setup: mode chips under the title, start/goal poster cards dead center, ALL rule knobs behind a rules pill with a live summary (`sceneRulesSummary`) opening `#scene-rules-sheet`. Knowledge mode hides classic-only bits via a `k-mode` class toggled on BOTH `#screen-scene` and `#scene-rules-sheet` (generic `.k-mode .only-classic`/`.only-knowledge` rules). The editor edits a *draft* (the loose `builder.mode/start/goal/...` fields); `builder.editing` is the scene index (-1 = new; new scenes auto-roll a matchup). Save Scene is disabled until the matchup is complete; ← back diffs against `sceneSnapshot0` and confirms before discarding a dirty draft. Hints/types are per-scene; only bans are global (`#rules-sheet`, applied live, voids par). Test Screening plays the feature with `builderTest = true`; finishing stamps `builder.par`/`builder.parRounds[]`; any material change calls `voidPar()`. Quitting a test run must reset `builderTest` AND `quest.active`.

**The casting call** (`screen-cast`) is the "browse big" tier — dropdowns stay the default everywhere; in-game search never uses it. Entered via `.browse-cast` buttons on endpoint cards/deck or the home switcher ("The Back Lot"). `cast.ctx` carries the context: `{mode:"slot", slot}` or `{mode:"slot", goal:i}` (tap fills that slot, returns to setup), `{mode:"studio", target}` (casting session — target chips Start/Goal/Via/Ban assign-and-stay, assigned cards lit via `updateCastMarks`/`data-role`, second tap un-casts via/bans, ✓ done re-renders the editor), `{mode:"research"}` (the Back Lot — tap opens `#detail-sheet`: credits as walkable cards, breadcrumb `cast.trail`, "Play from here" seeds a classic game). Grid cards keep RAW TMDB results (`cast.results`) because fame badges need `vote_count`/`popularity` and people show `known_for`; `makeItem` only on assign. Empty query falls back to `/trending`; typed-endpoint results lack `media_type`, so resolve via `raw.media_type || cast.type`. `cast.seq` guards stale responses.

**The Box Office** (`screen-office`, ghost button on home) shows feature shelves: `STAFF_PICKS` (curated v3 blobs bundled in app.js), Your Filmography (localStorage `"shelf"` — auto-saved by Studio Release/Print via `saveToShelf(buildStudioBlob())`, deduped, cap 30), Ticket Stubs (localStorage `"stubs"` — written by `recordStub()` from every quest ending; skips quick play and `builderTest`; stubs carry the blob for exact replays). Naming reservations: "Now Showing" = the future daily; "The Charts" = the future community shelf.

**Tickets** — a feature as an image file: `printTicket()` (Studio) draws a movie-ticket PNG on canvas and embeds the blob in a PNG `tEXt` chunk (`pngAddText`/`pngReadText`, keyword `connection-game`, hand-rolled CRC32). Redeem via the Box Office button or by dropping a PNG anywhere on the app (`redeemTicket` → `loadFeature`). Metadata-only by decision: the ORIGINAL file is required (recompression to JPEG strips the chunk); the stub barcode is decorative until a v2 QR.

**Settings** (`settings` object, localStorage `"settings"`): difficulty presets bundle obscurity/hints/timer; hand-tweaking any preset knob flips difficulty to `"custom"`. `hybridN` (double-feature goals) lives outside the presets. Difficulty/obscurity changes reroll every *visible* card via `rerollVisibleSlots()`.

## UI conventions

- Layout stability is a hard requirement: panels that appear/disappear reserve their space or scroll internally — the page must never grow and pop a scrollbar when toggling options.
- "Click depth = control depth" (Escher's principle): defaults need the fewest clicks; power hides behind pills/sheets/extra taps (rules pill, casting page, knobs reveal).
- Theming via CSS custom properties on `:root` / `body.dark`; any new color must work in both themes.
- Escape all TMDB-sourced strings with `esc()` before injecting into HTML.
- Async UI writes need stale-response guards (the `rerollSeq`/`buildRollSeq`/`cast.seq`/`detailSeq` token pattern) — follow it for anything that awaits then renders.
- Animations: `prefers-reduced-motion` must be respected (poster rain, view transitions already do).
- The design language is "warm paper cinema": grain texture (`--grain`), pill buttons/chips, `Instrument Serif` italics for accent words (`<em>`), gold for path highlights, sage `--start` / clay `--goal` for endpoints.
